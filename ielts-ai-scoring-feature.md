# 雅思写作 AI 智能辅助功能需求文档 (Feature Requirements)

## 1. 核心功能描述：智能补全 (Auto-complete)

类似代码补全（如 GitHub Copilot）的体验。在用户输入作文的过程中，AI 会根据当前题目（Topic）和已输入的上下文（Context），实时预测并生成后续的句子或段落内容。

### 1.1 交互体验

- **显示形式**：预测的补全内容以\*\*半透明、颜色较浅（Ghost text）\*\*的字体紧跟在光标后显示。
- **采纳方式**：用户按下 `Tab` 键即可接受建议，将补全内容转化为实际输入的文本。
- **触发时机**：在用户打字停顿、或者写完一个词/短语时自动触发预测。

### 1.2 内容生成标准 (Band 7+ Standard)

- **词汇与句型**：所有补全的内容必须符合**雅思 7 分及以上**的评分标准，使用地道的词汇搭配（Collocations）、多样的句型结构（Complex sentences）和恰当的连接词（Cohesive devices）。

### 1.3 结构化约束

补全内容必须严格遵循雅思写作的段落结构规范，AI 需具备感知当前写作段落的能力：

#### Task 1 (小作文) - 四段式结构

1. **题目改写段 (Introduction)**：Paraphrase 题目，介绍图表描述的内容。
2. **总体概述段 (Overview)**：总结图表最显著的趋势、差异或主要特征（无数据细节）。
3. **细节段 1 (Body Paragraph 1)**：描述第一组相关的详细数据和特征。
4. **细节段 2 (Body Paragraph 2)**：描述剩余的详细数据和特征，并进行对比。

#### Task 2 (大作文) - 五段式结构

1. **开头段 (Introduction)**：背景引入，改写题目，并清晰表明总观点/立场（Thesis Statement）。
2. **主体段 1 (Body Paragraph 1)**：第一个分论点及展开（解释、举例、结果）。
3. **主体段 2 (Body Paragraph 2)**：第二个分论点及展开。
4. **主体段 3 (Body Paragraph 3)**：第三个分论点及展开（或让步反驳段）。
5. **结尾段 (Conclusion)**：重申观点，总结主体段核心理由，不提出新观点。

## 2. 核心功能描述：实时语法纠错 (Real-time Correction)

在写作过程中，除了预测未来，AI 还会实时对用户刚刚写完的内容进行审查。

### 2.1 交互体验

- **触发时机**：当用户**完成当前句子的输入**（例如输入句号、问号等标点符号并停顿时）触发当前句的检查。
- **显示形式**：
  - 在有语法错误或表达不地道的原文上，直接添加**删除线 (Strikethrough)**。
  - 紧接着在后方，以**明显的样式（如绿色字体或特定的高亮背景）**显示正确的替代表达。
- **目标**：只纠正基础的错误（如语法、时态、拼写等），不再负责低级词汇的高级替换。

## 3. 新增核心功能：划词润色与重写 (Task C: Selection Polish)

为了提供更深度的写作辅导，新增了基于鼠标选中的划词润色功能。

### 3.1 交互体验

- **触发时机**：用户使用鼠标在编辑器中选中一段文本后，自动触发。
- **展现形式**：在选中文本附近弹出一个小浮窗（Popover/Tooltip）。弹窗内的建议支持点击一键替换。
- **处理逻辑分类**：
  1.  **选中单个单词**：AI 在弹窗中提供 1-3 个符合语境的**同义高级替换词（雅思 7 分以上级别）**。
  2.  **选中单个句子**：在不改变当前句子核心意思的前提下，AI 采用其他高级的句式结构（如从句、高级连接词、倒装/虚拟语气等）来替代原句。
  3.  **选中多个句子**：AI 使用正确合理的逻辑关系和复杂句式，将这几个短句合并成一句高度凝练的长难句。

## 4. 技术实现挑战与架构考量 (技术实现指南)

如果确定按照以上需求开发，我们在技术栈上需要做以下重大调整：

1. **替换基础文本框为富文本编辑器 (Rich Text Editor)**：
   - 目前的 `<textarea>` 无法实现同一文本框内显示“删除线”、“绿色字体”以及“半透明幽灵文本（Ghost text）”。
   - **解决方案**：需要引入支持自定义装饰（Decorations）和按键拦截（Key binding）的编辑器，例如 **ProseMirror**, **Slate.js**, 或者常用于代码编辑器的 **Monaco Editor** / **CodeMirror 6**（非常适合实现 Copilot 效果）。
2. **AI 模型的流式传输 (Streaming API)**：
   - 打字补全对延迟极其敏感。我们不能像现在的全文批改一样等 10 秒钟。必须使用 OpenAI/DeepSeek 接口的 `stream: true` 参数，边生成边在光标后渲染。
   - 需要实现请求防抖（Debounce）和请求中断（AbortController），在用户继续打字时取消上一次的旧预测。
3. **三轨 Prompt 设计**：
   - 我们需要三个并行的 AI 任务：
     - **Task A (Autocomplete / Band 7 Candidate Scaffolding)**：系统提示词需要强调“你是一名目标分数为雅思7分的考生。在用户写完一句话的结尾时，**不要**直接预测下一句的开头。必须等待用户自己写出新句子的开头（3-5个词）之后，再顺着用户的思路给出符合7分习惯的短语骨架或地道搭配（Collocations）。你的所有输出必须基于对题目的理解，直接输出简短的续写内容。”
     - **Task B (Correction)**：系统提示词要求“审查最后输入的这句话。仅当存在明显的语法、时态或拼写等基础错误时，返回 JSON 格式的修改建议：原文、修改后。不需要提供高级表达，不需要提供错误原因。” 前端会将错误部分划上红色删除线，正后方用绿色字体显示正确表达，用户按下 `Tab` 键即可采纳。
     - **Task C (Selection Polish)**：系统提示词要求“你是一个雅思高级润色引擎。如果用户输入的是单个单词，提供1-3个同义高级替换；如果是单句，使用高级句式重写；如果是多句，使用高级逻辑连词合并为一句长难句。保持原意不变。”

## 4. My Collections (历史记录) 页面体验升级

为了提升用户复盘和学习的体验，对 `CollectionsPage.tsx` 中的历史记录展示卡片进行以下改进：

### 4.1 标题可编辑

- **需求**：目前卡片的标题只是简单的 "Image Topic" 或文本题目的前几个字，缺乏辨识度。
- **实现**：允许用户点击标题 `p` 标签进行就地编辑（Inline Edit），并同步保存到 `localStorage` 中，方便用户根据自己的习惯命名（例如：“2024.1 柱状图练习”）。

### 4.2 图片放大与题目识别

- **需求**：对于上传了图片的题目，目前的 `img` 缩略图太小，且没有直接展示题目文字。
- **实现**：
  1. 点击图片缩略图时，支持弹出全屏大图（Lightbox / Dialog 放大预览）。
  2. 不做图片题目文字提取，仅提供放大查看能力。

### 4.3 双师联合批改系统 (Dual-Persona Feedback)

- **需求**：将原本单一的整体评价 `<p>` 拆分为两位具有不同侧重点的虚拟教师点评，增强互动感和专业度。
- **实现机制**：修改打分接口（`scoreEssay`）的 Prompt，要求大模型输出两个不同角色的点评内容，并在前端结合 AI 生成的卡通头像展示：
  - 👩🏼 **Kelly（外教老师）**：
    - **形象**：金发碧眼、年轻且具有亲和力的经典白人女性英语母语者。
    - **点评侧重**：从语法用词（GRA/LR）以及语言是否符合母语者使用习惯的角度给出评价，指出不地道的表达。
  - 👨🏻‍🏫 **杰明/Jieming（中教老师）**：
    - **形象**：黑发戴眼镜、年轻专业且具有亲和力的中国英语教师。
    - **点评侧重**：从文章结构（CC）、逻辑连贯性、是否切题（TR）并符合雅思官方评分标准的角度给出评价。

### 4.4 高分重写范文格式优化 (Rewritten Essay Formatting)

- **需求**：目前底部的重写范文没有分段，且难以一眼看出好词好句。
- **实现**：
  1. **强制分段**：修改 Prompt，严格要求 AI 在生成的范文中使用正确的换行符（`\n\n`）或 Markdown 格式进行段落划分。
  2. **亮点加粗**：要求 AI 在范文中，将所有**高级词汇、地道搭配、以及复杂的出彩句型**直接使用 Markdown 加粗语法（`**bold**`）标出。
  3. **前端渲染**：前端使用 `ReactMarkdown` 或类似组件正确渲染这些段落和加粗样式。
